# Security Report (Jafar) — Design Spec

**Date**: 2026-04-23
**Branch**: `feature/security-report-jafar` (to be created)
**Audience**: Jafar (role ID 6) only
**Status**: Approved for planning

## Overview

A Jafar-only page that renders the latest `gstack cso` security audit report from `.gstack/security-reports/*.json` and lets Jafar operators review historical scans.

The report already exists as structured JSON (attack surface census, 18 findings with severity/confidence/exploit-scenario/recommendation, supply-chain summary, trend vs prior scan). This project surfaces it inside Guardian instead of requiring operators to open the raw file, and adds a history dropdown so scans can be compared over time.

The reports contain live exploit payloads for unpatched vulnerabilities in the running app. The page is Jafar-only and the underlying directory is never shipped to production — see **Security & Ops Safety Rails**.

### Interpreting "add a Jafar permission level in the settings"

The existing Jafar role (role ID 6) already exists and is already enforced by `getAuthenticatedUserCompany + checkJafarRole` server-side and `RequireJafar` client-side. We are **reusing** that role, not creating a new one. "In the settings" is interpreted as "as a new Jafar-gated entry in the Admin Dashboard's Jafar section," consistent with the four Jafar tiles already there (Hard Delete, Users, Site Analysis, Role Access Matrix). If the intent was the Account Settings page instead, that's a different spec and we should stop here.

## Scope

### In scope (MVP)

- Jafar-only React page at `/jafar/security-report`, wrapped in `ProtectedRoute + RequireJafar`
- New Admin Dashboard card (5th Jafar tile) matching the existing Jafar toolkit UX
- Three read-only JSON endpoints under `/api/jafar-admin/security-reports/*` (added inline in all three legacy server files)
- Report history dropdown (newest first), defaulting to the most recent scan
- Summary cards (`totals` + `trend.new`), collapsible Attack Surface and Supply Chain panels, filterable Findings list
- Per-finding "Copy recommendation" and "Copy playbook" buttons
- Expand all / Collapse all controls; severity filter chips + free-text search over title/file/category
- Browser Print / Save-as-PDF button
- Safety rails: directory never deployed, path-traversal-proof filename validation, graceful empty state when dir missing

### Explicitly out of scope (deferred)

- Diff view of a report vs a prior scan (the JSON already carries a `trend` block; can layer on later using `trend.prior_report_date`)
- Bespoke PDF export (browser Print is enough for a CISO review)
- "Mark finding as resolved" UI — reports are regenerated by `gstack cso`, not hand-edited
- Any write endpoints — read-only surface
- Linking findings to Jira/Linear tickets — not requested
- Email / Slack notifications on new Critical findings — not requested

## Design decisions (locked in during brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Historical reports | Dropdown, newest selected by default, no diff view yet | Low complexity, JSON already carries `trend` so a diff view is additive later |
| URL prefix | `/api/jafar-admin/security-reports/*` | Matches the existing `/api/jafar-admin/*` convention (users, companies, purge, jafar-users) |
| Server placement | Inline in `server.cjs` / `server.js` / `server-production.js` | The TypeScript `server/routes/` tree is unmounted; all live Jafar routes are inline. Same pattern as `server.cjs:15455+` |
| Middleware | `getAuthenticatedUserCompany, checkJafarRole` | Matches every other `/api/jafar-admin/*` route |
| Base directory | `path.resolve(process.cwd(), '.gstack/security-reports')` | Same location where `gstack cso` writes reports; not shipped to prod |
| Production behavior | Endpoints return empty list / 404 when base dir missing | `.gstack/` never deployed — reports don't exist in prod, and that's correct |
| Dashboard placement | 5th Jafar tile on `AdminDashboard` | Mirrors existing JAFAR tile pattern |
| Caching | None | Files are small (~35KB), access is rare (Jafar-only) |

## Architecture

### Codebase reality check (informs decisions below)

- All live Jafar routes are inline in the three legacy server files under `/api/jafar-admin/*`, using `getAuthenticatedUserCompany, checkJafarRole` (confirmed at `server.cjs:15455`). The `server/routes/jafar-admin.ts` tree exists but is not mounted — adding files there has no runtime effect. New routes therefore go inline in all three legacy server files, in sync.
- The project has no test runner installed — smoke tests are written as standalone TS files under `src/tests/` and `server/tests/`, consistent with `src/tests/site-analysis.smoke.test.ts` and `src/tests/login-tracking.smoke.test.ts`. Manual verification is the primary QA gate.
- `.gitignore` does **not** currently ignore `.gstack/`. The one existing report (`2026-04-23-070810.json`) is tracked in git. The spec does not scrub history — that's a separate decision captured in the existing security report Finding #1 (secrets exposure) and belongs in a remediation effort, not this feature.

### New surface

**Backend (inline in all three legacy server files, in sync):**

- `GET /api/jafar-admin/security-reports` — lists available reports, newest first, metadata only (`filename`, `date`, `mode`, `scope`, `totals`, `trend`). Used by the history dropdown.
- `GET /api/jafar-admin/security-reports/latest` — full JSON of the newest report.
- `GET /api/jafar-admin/security-reports/:filename` — full JSON of a specific report. `:filename` is validated against `/^\d{4}-\d{2}-\d{2}-\d{6}\.json$/` and the resolved path must start with the base directory.

All three use the existing middleware chain: `getAuthenticatedUserCompany, checkJafarRole`.

**Frontend:**

- `src/pages/SecurityReport.tsx` — page component. Loads report list on mount, auto-selects newest, lazy-loads the full JSON when a report is picked.
- `src/components/SecurityReport/SummaryCards.tsx` — four stat cards (Critical / High / Medium / New since last scan).
- `src/components/SecurityReport/AttackSurfacePanel.tsx` — collapsible panel for `attack_surface.code` and `attack_surface.infrastructure`.
- `src/components/SecurityReport/SupplyChainPanel.tsx` — collapsible panel for `supply_chain_summary`.
- `src/components/SecurityReport/FindingCard.tsx` — single finding, collapsed by default. Severity badge (colored), confidence `X/10`, status badge, title, `file:line` (monospace), category. Expandable body: description → exploit_scenario → impact → recommendation → playbook → verification. "Copy recommendation" and "Copy playbook" buttons in the expanded view.
- `src/components/SecurityReport/FindingsToolbar.tsx` — severity filter chips (ALL / CRITICAL / HIGH / MEDIUM), free-text search over title/file/category, "Expand all" / "Collapse all" buttons.

**Shared types** (`src/types/securityReport.ts`): TypeScript interfaces for `SecurityReport`, `SecurityFinding`, `AttackSurface`, `SupplyChainSummary`, `Totals`, `Trend`. Derived from the actual v2.0.0 payload.

**Route wiring:**

- `src/App.tsx` — add `<Route path="/jafar/security-report" element={<ProtectedRoute><RequireJafar><SecurityReport /></RequireJafar></ProtectedRoute>} />`
- `src/pages/AdminDashboard.tsx` — add a 5th Jafar tile, same visual pattern as the existing "Role Access Matrix" tile (title, 3 bullet points, Jafar-only conditional render).

### Data flow

1. User lands on `/jafar/security-report`.
2. Page fires `GET /api/jafar-admin/security-reports` — gets metadata list.
3. Page auto-selects entry `[0]` (newest) and fires `GET /api/jafar-admin/security-reports/latest` to hydrate the full payload.
4. Switching the dropdown fires `GET /api/jafar-admin/security-reports/:filename` for that specific report and re-renders.
5. Severity chips and the search input filter the findings list client-side against the already-loaded payload.
6. Expand/collapse is local component state.
7. "Copy recommendation" / "Copy playbook" use `navigator.clipboard.writeText` with a toast confirmation.

## Backend handler specifics

All three handlers share one helper module (duplicated across the three legacy server files, in sync):

```js
const SECURITY_REPORTS_DIR = path.resolve(process.cwd(), '.gstack/security-reports');
const REPORT_FILENAME_RE = /^\d{4}-\d{2}-\d{2}-\d{6}\.json$/;

async function listReportFilenames() {
  try {
    const entries = await fs.promises.readdir(SECURITY_REPORTS_DIR);
    return entries
      .filter(name => REPORT_FILENAME_RE.test(name))
      .sort()                    // ascending
      .reverse();                // newest first
  } catch (err) {
    if (err.code === 'ENOENT') return [];
    throw err;
  }
}

async function readReport(filename) {
  if (!REPORT_FILENAME_RE.test(filename)) {
    const e = new Error('INVALID_FILENAME');
    e.status = 400;
    throw e;
  }
  const resolved = path.resolve(SECURITY_REPORTS_DIR, filename);
  if (!resolved.startsWith(SECURITY_REPORTS_DIR + path.sep)) {
    const e = new Error('PATH_TRAVERSAL');
    e.status = 400;
    throw e;
  }
  const raw = await fs.promises.readFile(resolved, 'utf8');
  return JSON.parse(raw);
}
```

- `GET /api/jafar-admin/security-reports` → `listReportFilenames()`, read each file, return `{ reports: SecurityReportSummary[] }`. Files that fail to parse are logged and skipped, not fatal.
- `GET /api/jafar-admin/security-reports/latest` → first filename from the list, or `404 { error: 'No reports available' }` when list is empty.
- `GET /api/jafar-admin/security-reports/:filename` → `readReport(req.params.filename)`. Invalid regex → `400`. Missing file → `404`.

Handlers are registered alongside the existing Jafar routes at `server.cjs:~15455`, and copy-synced to `server-production.js` and `server.js` per the project's multi-server synchronization protocol.

## Error, empty, and loading states

| State | Trigger | UX |
|---|---|---|
| Loading list | Initial mount | Skeleton for summary cards + findings list |
| Loading report | Dropdown change | Inline spinner in the header, body stays as last-rendered until new payload arrives |
| No reports available | List endpoint returns `{ reports: [] }` | Friendly empty state: "No security reports available on this environment. Run `gstack cso` locally to generate one." |
| Fetch error | Network / 5xx | Toast (`toast.error`) + inline "Retry" button |
| Malformed report | `:filename` endpoint returns 400/404 | Toast + revert dropdown to previous selection |
| Non-Jafar user | API returns 403 | `RequireJafar` component already redirects before the page mounts |

## Security & Ops Safety Rails (mandatory)

1. **`.gstack/` never ships to production.** Confirm the directory is not copied by `azure-pipelines.yml` / `azure-pipelines-staging.yml` and that `web.config` doesn't expose it. If the pipeline uses a broad `cp -R` that would include it, add an explicit exclusion. If not, add a comment documenting the requirement so a future dev doesn't accidentally include it.
2. **`.gitignore` entry for `.gstack/`** — currently missing. Add it as part of this feature so future scan artifacts are not committed. Leave the already-tracked `2026-04-23-070810.json` in place; git-history scrubbing is a remediation task for the secrets-exposure finding (#1 in the report), not this feature.
3. **Filename validation** — strict regex `^\d{4}-\d{2}-\d{2}-\d{6}\.json$` + resolved-path prefix check on every request touching the filesystem. This is defense-in-depth against the same path-traversal class as Finding #3 in the report itself.
4. **Middleware order** — handlers are mounted *after* `getAuthenticatedUserCompany, checkJafarRole`, never before.
5. **No params feed the filesystem path** beyond `:filename` validated above. No `?path=`, no `?dir=`.
6. **Production response shape** — when `.gstack/` does not exist on disk, the API responds with `{ reports: [] }` (list) or `404` (latest/by-filename), never 500. The page then renders the friendly empty state.
7. **Never log the report JSON** to stdout/stderr. The payload includes live exploit payloads — logs go to Log Analytics and Finding #10 already flags log exfil as a risk.

## Testing

The project has no test runner installed, so tests follow the existing `.smoke.test.ts` convention (standalone TS files runnable by hand) and manual verification is the primary gate.

### Automated smoke tests

- `server/tests/security-reports.smoke.test.ts`
  - Lists reports sorted newest-first
  - Rejects `../` / `..%2F` / arbitrary non-matching filenames with 400
  - Returns `{ reports: [] }` when base dir absent (simulated via temp dir)
  - Returns 404 from `/latest` when list is empty
- `src/tests/security-report.smoke.test.ts`
  - Severity filter narrows the rendered finding list
  - Dropdown change triggers a re-fetch against `/:filename`
  - Empty-state message renders when `reports: []`

### Manual verification checklist

1. Log in as a Jafar user locally → Admin Dashboard shows the new "Security Report" card.
2. Open the page → summary cards show 6/9/3/18, findings list populated, severity filters work, search narrows the list.
3. Expand a finding → description, exploit scenario, impact, recommendation, playbook, verification all render with clear labels.
4. Click "Copy recommendation" → toast confirms, clipboard has the recommendation text.
5. Click "Copy playbook" → toast confirms, clipboard has the playbook text.
6. Open Print dialog → page renders in a readable print layout.
7. Log in as a non-Jafar user → card is hidden; direct navigation to `/jafar/security-report` is blocked by `RequireJafar`; direct API call to `/api/jafar-admin/security-reports` returns 403.
8. Rename `.gstack/security-reports/` locally → page shows the "No security reports available" empty state, no error toast.
9. Craft a `GET /api/jafar-admin/security-reports/..%2F..%2F.env` — confirm 400.
10. Verify `.gstack/` is ignored and not in the deployment zip produced by the staging pipeline.

## Risks

| Risk | Mitigation |
|---|---|
| Report JSON accidentally deployed to production | `.gitignore` entry, pipeline audit, API returns empty when dir missing so nothing leaks |
| Path traversal via `:filename` | Strict regex + resolved-path prefix check + unit tests covering `../`, `%2F`, absolute-path attempts |
| Log exfil of exploit payloads | Handlers never log the parsed JSON body |
| Schema drift as `gstack cso` evolves | Frontend is defensive: unknown severities render neutral, missing optional fields render gracefully. `version` is displayed in the header so schema bumps are visible |
| Server file drift between `server.cjs` / `server.js` / `server-production.js` | Multi-server synchronization protocol already in place (CLAUDE.md); spec calls out all three files explicitly |
