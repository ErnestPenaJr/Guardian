Approach B: Design System First, Pages Second

Build a complete component library first, then apply it everywhere. More infrastructure upfront, but highly systematic and future-proof.

How Approach B Differs
Instead of reskinning pages directly, you first build a complete component library with every primitive (Modal, Sidebar, Table, EmptyState, StatusBadge, ProgressStepper, Tooltip, etc.). Each component is documented in a living StyleGuide. Then pages are rebuilt using only those components — no raw HTML or one-off styles allowed.

Approach B Trade-offs

Pros

  • Highly systematic — every component documented and reusable
  • Future-proof — new pages auto-inherit the design system
  • Enforced consistency — no one-off styles possible
  • Onboarding — new devs learn from the StyleGuide

Cons

  • Slower visible results — Phase 1 is infrastructure, no user-facing changes
  • Two-phase rollout — inconsistency persists during Phase 1
  • Over-engineering risk — may build components you don't actually need
  • More total effort — documentation + components + page migration
